## The Component-Based Structure Model
This is a strong direction. The key insight: you don't need spatial complexity (Minecraft's block grid), just **a list of parts that each contribute to computed stats**.
```csharp
public class StructureComponent
{
    public ComponentType Type { get; set; } // BranchWall, HideWall, StoneFoundation, etc.
    public float Condition { get; set; } // 0-1, degrades over time
   
    // Derived from Type + Condition
    public float WindCoverage => _baseWindCoverage * Condition;
    public float Insulation => _baseInsulation * Condition;
    public float Durability => _baseDurability * Condition;
}
public class Shelter
{
    public List<StructureComponent> Components { get; }
   
    // Computed from all components
    public float EffectiveWindCoverage => Math.Min(1f, Components.Sum(c => c.WindCoverage));
    public float EffectiveInsulation => Math.Min(1f, Components.Sum(c => c.Insulation));
    public float StructuralIntegrity => Components.Average(c => c.Durability);
}
```
A shelter isn't binary. It's:
- 3 branch wall sections (condition: 0.8, 0.6, 0.4)
- 1 hide wall section (condition: 1.0)
- 2 branch roof pieces (condition: 0.7, 0.5)
- 1 hide insulation layer (condition: 0.9)
**What this gives you:**
| Dynamic | How It Emerges |
|---------|----------------|
| Incremental improvement | Add one component per work session |
| Granular damage | Storm damages specific components, not "the shelter" |
| Partial repair | Fix the worst components first, or the most critical |
| Resource tradeoffs | Use hide for wall (permanent) or save for clothing? |
| Hodgepodge aesthetic | The component list *is* the description |
| NPC contribution visible | "While you were gone, Kira added a branch wall section" |
## Component Types and What They Contribute
For shelters:
| Component | Wind | Overhead | Insulation | Durability | Material Cost |
|-----------|------|----------|------------|------------|---------------|
| Branch Wall | 0.08 | — | 0.02 | Low | 2 branches |
| Hide Wall | 0.12 | — | 0.06 | Medium | 1 hide |
| Snow Wall | 0.15 | — | 0.10 | Degrades in warmth | Snow + time |
| Branch Roof | — | 0.15 | 0.02 | Low | 3 branches |
| Hide Roof | — | 0.20 | 0.08 | Medium | 1 hide |
| Stone Foundation | — | — | 0.03 | High | 3 stone |
| Hide Insulation | — | — | 0.12 | Medium | 1 hide |
Caps at 1.0 for coverage stats. So a "complete" shelter might need 6-8 wall sections and 4-5 roof pieces. But you can survive with less—it's just worse.
**The same pattern applies to other structures:**
**Fire Pit:**
- Stone ring sections → wind protection, heat retention
- Reflector wall → directional heat bonus
- Cooking stone → enables cooking without tools
- Ash bed → ember preservation bonus
**Cache:**
- Depth (dug out) → base capacity
- Stone lining → rodent protection
- Hide covering → weather protection
- Raised platform → predator protection
- Each adds capacity and/or protection rating
**Drying Rack:**
- Posts → base capacity
- Crossbars → additional capacity
- Hide awning → weather protection (but slower drying)
## Degradation and Damage
Weather and time wear components:
- Base degradation rate per day (slow)
- Storm events accelerate degradation
- Low-durability materials (branch) degrade faster than high (stone, hide)
- Below-freezing + wet = faster wood degradation
Damage targeting:
- Random component selection, weighted toward low-condition (weak points fail first)
- Severe events can destroy components entirely (condition → 0, removed)
- This creates ongoing maintenance pressure without binary "shelter destroyed" moments

## Keeping It Simple
The implementation is just:

- Component types as enum with static property tables
- Computed stats via LINQ aggregation
- Weather/time ticks reduce condition
- Work options: AddComponent, RepairComponent
You're not simulating individual boards. You're simulating *categories of improvement* that stack. The player's imagination fills in what "3 branch walls and a hide wall" looks like. The mechanical reality is just: numbers that add up to shelter effectiveness.

## Ideas That Actually Matter (Not Generic)
### 1. Event Memory With Callbacks
NPCs don't just track relationship values—they remember **specific events** with tagged emotional valence.
```
Memory:
    Event: PlayerSharedFoodWhenStarving
    Tick: 4320
    Valence: +0.4
    Referenced: false
```
When similar contexts arise, there's a chance the NPC references the memory:
- You share food again → "Kira takes it without hesitation. She remembers the last time."
- You're starving and Kira has food → She shares immediately (memory triggers reciprocity), toast: "Kira hands you dried meat. Doesn't even hesitate."
- Kira is about to die → "Kira looks at you. The same look from when you shared your last food."
The memory doesn't have to be complex. Just: who, what category, when, whether it's been referenced yet. The narrative weight comes from *surfacing* it at the right moment.
**What this costs:** A small list per NPC, event categorization, trigger conditions for callbacks.
**What it creates:** The feeling of shared history. "Remember when" moments that emerge from play.
---
### 2. Visible Interiority Through Behavioral Tells
Personality traits shouldn't be invisible stats—they should manifest as **observable behavioral patterns** the player can learn to read.
| Trait | Behavioral Tell |
|-------|-----------------|
| High Selfishness | Hovers near resources. Eats first. Positions themselves closer to fire center. |
| Low Selfishness | Waits for others to eat. Gives up fire position. Carries more than their share back. |
| High Boldness | Ranges far. Faces threats instead of fleeing. Sleeps facing outward. |
| Low Boldness | Stays close to camp. Startles at sounds. Sleeps near center, facing inward. |
| High Sociability | Always near others. Checks on people. Follows without being asked. |
| Low Sociability | Keeps distance. Takes solo work. Positions themselves at edge of camp. |
This isn't about new mechanics—it's about **spatially expressing** the mechanics you already have. If the tile map shows NPC positions, a selfish NPC being consistently closer to the fire tells you something without any UI tooltip.
**Extension:** Mood modifies tells. A normally bold NPC who becomes timid after a bad encounter reads as traumatized. A normally generous NPC who starts hoarding reads as scared.
---
### 3. Asymmetric Relationships Create Drama
Right now you're simplifying to one axis. Fine. But make sure **A→B ≠ B→A**.
The player might be trusted by Kira but not trust her back. Kira keeps following them, helping them, saving resources for them—and the player keeps taking without reciprocating. Eventually Kira's trust decays from asymmetry. Or doesn't, and her loyalty becomes poignant.
The inverse: the player trusts an NPC who doesn't trust them. They keep trying to help, and the NPC keeps rejecting it, staying distant. Earning that trust becomes a goal.
**This matters for NPC-NPC relationships too.** If you add more NPCs later, two NPCs with mismatched relationships (A loves B, B tolerates A) creates visible dynamics without any authored content.
---
### 4. The Ask System (NPCs Request Things)
Your suggestion system is player→NPC. Flip it.
NPCs with sufficient trust **ask the player for things:**
- "Kira gestures toward the eastern ridge. She wants to scout it. Waiting for your response."
- "Mira is shivering. Looks at your extra hide meaningfully."
- "Theron has been watching the wolf tracks. He wants to hunt."
The player can:
- Agree (relationship boost, NPC acts on it)
- Ignore (small relationship decay over time if repeated)
- Refuse (immediate small hit, but at least you responded)
**Why this matters:** It flips agency. NPCs stop being tools and start being agents with desires. The player has to consider them, not just deploy them.
**Trust threshold for asking:** Low-trust NPCs don't ask—they just suffer or leave. Only NPCs who trust you will make themselves vulnerable by requesting.
---
### 5. NPCs Name Things
When an NPC witnesses something significant at a location, they might name it:
- Kira kills a wolf at location (4,7) → "Wolf Rock" (Kira's name)
- Someone nearly dies from cold at a location → "The Cold Place"
- A cache is found → "Lucky Hollow"
Names appear on the map. Different NPCs might have different names for the same place based on their experience.
**Why this matters:** The world stops being generic tiles and becomes *places with history*. This is the text-game equivalent of environmental storytelling.
---
### 6. Death Weight Through Specificity
When an NPC dies, the *how* matters:
| Death Context | Effect on Survivors |
|---------------|---------------------|
| Died saving someone | Savee gets permanent +0.2 to all relationships, grief effect, strong memory |
| Died alone (no witnesses) | Discovery triggers grief. "You find Kira's body. The cold took her." |
| Died while player could have helped but didn't | Witnesses get suspicion. Trust drops. Memory: "Stood by while Kira died." |
| Died in combat alongside others | Combat survivors bond. Shared memory of the fight. |
| Died due to someone's negligence | Blame attribution. "The fire went out on your watch." |
**The body matters.** An NPC body should be a feature on the tile—not just a loot container, but something that persists. Other NPCs might visit it. You can bury it (removes feature, grants closure, relationship boost with anyone who cared about them).
---
### 7. Trust Decay From Asymmetric Investment
If an NPC consistently gives more than they receive, trust should *slowly* decay even without negative events. They're being taken for granted.
```
if (npc.RecentContributions[target] > target.RecentContributions[npc] * 1.5)
    relationshipDelta -= 0.01 per day
```
This creates **maintenance relationships.** You can't just save someone once and coast forever. You have to keep demonstrating that you value them.
**The flip side:** If someone contributes nothing but keeps receiving, resentment builds in contributors. This could drive NPC-NPC conflict if one NPC is parasitic.
---
### 8. Departure Instead of Death
Low-trust NPCs who aren't in critical need **can leave.**
Conditions:
- Trust < threshold (maybe -0.2)
- Not in survival crisis
- Has been at camp long enough to have formed opinion
- Nothing forcing them to stay
They don't just disappear—they announce it:
"When you wake, Theron is gone. His bedding is cleared. He took his share of supplies—nothing more."
**Why this matters:** It creates stakes for relationship maintenance that aren't just "they won't help." They'll *leave*. And if NPCs are rare (not infinitely spawning), losing one is permanent.
---
### 9. Joint Activities That Require Trust
Some activities are only possible or much better with trusted cooperation:
| Activity | Solo | With Untrusted NPC | With Trusted NPC |
|----------|------|-------------------|-----------------|
| Sleep | Normal | Can't (won't turn back) | Watches while you sleep, better rest |
| Hunt large game | Dangerous | They flee at first sign of danger | Coordinates, flanks, rescues if down |
| Travel far | Risky | They might abandon you if things go bad | Will carry you back if injured |
| Build shelter | Slow | Uncoordinated, slow | Faster, better quality |
This makes trust *functional*, not just social. A trusted ally is mechanically different from a stranger tagging along.

### 10. Specialization Through Repetition (Not Stats)
NPCs don't have skills. But they have **experience memory**:
```
ExperienceCounts:
    FireStarting: 47
    Hunting: 12
    Foraging: 89
```
High experience in an activity could mean:
- Faster completion (diminishing returns)
- Better yields (slight bonus)
- Preference (they choose this when idle)
- Identity (other NPCs and player see them as "the forager")
An NPC who has started 50 fires is "the fire-keeper." They gravitate toward fire-tending. They're better at it. Losing them means losing the camp's fire competence.
**This isn't skill points.** There's no UI saying "+2 Fire." It's just observed behavior + slight mechanical edge that emerges from what they've actually done.

### 11. Fear Memory / Place Avoidance
NPCs remember where bad things happened to them:
```
FearMemory:
    Location: (4, 7)
    Event: WolfAttack
    Severity: 0.8
```
Effects:
- Avoid pathing through feared locations if alternatives exist
- If forced to go there, visible anxiety (toast: "Kira hesitates at Wolf Rock. Bad memories.")
- Boldness check to enter voluntarily
- Fear decays slowly, faster if they return and nothing bad happens
**This makes NPCs feel like they have a past.** The map isn't neutral to them—it's marked with their experiences.

### 12. Base Building Integration: Ownership and Pride
NPCs should have **ownership relationships** with structures they built:
- "Kira's shelter" (she built most of it)
- Building something grants attachment
- Damage to "their" structure bothers them more than general damage
- They prioritize maintaining their own structures
**Camp investment affects loyalty.** An NPC who has built a lot at camp is less likely to leave—they have roots. An NPC who just arrived and contributed nothing has no stake.
```
CampInvestment = sum of (component.value * built_by_me_factor)
```
High investment → harder to trigger departure, more grief if camp is destroyed/abandoned.

On the tile map, NPC emoji/icon should **reflect their current state:**
- Healthy, warm, fed → normal
- Cold → hunched/shivering indicator
- Hungry → hand-to-stomach
- Injured → limping/bandage
- Scared → small, withdrawn
- High trust with player → facing toward player
- Low trust → facing away
This lets you read NPC states at a glance without tooltips. The tile map becomes a social readout.
